# 06. 실제 실험 재현을 위해 추가로 필요한 것 (Gap 분석)

## 전제: PaperCoder는 "코드 생성"까지만 한다

가장 중요한 전제는, **PaperCoder는 논문을 실행 가능한 코드 초안으로 옮기는 것까지만 자동화**하고,
그 코드를 **실제로 실행해 논문의 실험 수치를 재현하지는 않는다**는 점입니다.

- 기본 파이프라인([`scripts/run.sh`](../scripts/run.sh))은 `0 → 1 → 1.1 → 2 → 3`, 즉 **코드 생성(3단계)에서 종료**됩니다.
- 논문의 평가([`codes/eval.py`](../codes/eval.py))는 실행 결과가 아니라 **LLM 심사자가 코드 충실도를 1~5점으로 채점**하는 방식입니다.

따라서 "논문 표의 수치까지 재현"을 목표로 한다면, 아래 항목들은 저장소가 채워주지 않으며 사용자가 보완해야 합니다.

---

## 1. 실행·디버깅 단계가 기본 흐름에서 빠져 있음

실제 실행에 필요한 두 스크립트는 `run.sh`에 연결돼 있지 않고 **별도로 실행**해야 합니다.

| 스크립트 | 역할 | 상태 |
|----------|------|------|
| [`3.1_coding_sh.py`](../codes/3.1_coding_sh.py) | 환경 생성·의존성 설치·엔트리포인트 실행을 담은 `reproduce.sh` **생성** | 기본 `run.sh`에 미포함 |
| [`4_debugging.py`](../codes/4_debugging.py) | 실행 에러 로그를 받아 SEARCH/REPLACE 패치를 적용 | 기본 `run.sh`에 미포함 |

즉 **"생성 → reproduce.sh 생성 → 실행 → 에러 수집 → 디버깅 → 재실행"** 루프는 사용자가 수동으로
오케스트레이션해야 합니다.

> ⚠️ **알려진 버그**: [`4_debugging.py:138`](../codes/4_debugging.py#L138)은 `args.output_repo_dir`를
> 참조하지만, 이 스크립트의 `argparse`에는 `--output_repo_dir`가 정의돼 있지 않습니다
> (정의된 인자: `--error_file_name`, `--output_dir`, `--paper_name`, `--model`, `--save_num`).
> 그대로 실행하면 `AttributeError`가 발생하므로, 디버깅 단계를 쓰려면 이 인자 추가/수정이 필요합니다.

---

## 2. 실제 학습 데이터셋은 제공되지 않음

- 생성 코드는 dataset loader를 만들지만, 프롬프트가 **"논문에 없는 값은 지어내지 말 것(DO NOT FABRICATE)"** 을
  강제하므로 실제 데이터 **다운로드/경로/전처리 로직은 비어 있거나 placeholder**입니다.
- 논문 실험에 쓰인 실제 데이터셋(예: WMT14, ImageNet 등)은 **직접 확보·연결**해야 합니다.
- 주의: [`data/paper2code`](../data/paper2code)에 있는 것은 학습 데이터가 아니라 **논문 텍스트(JSON) 벤치마크**입니다.

---

## 3. 논문에 명시되지 않은 하이퍼파라미터/디테일 공백

- `config.yaml`은 **논문에 명시된 값만** 담습니다.
- 나머지 불명확한 부분은 설계 산출물의 **`Anything UNCLEAR`** 항목으로 남습니다.
- 실제 수치 재현을 노린다면 이 공백들(초기화 seed, optimizer 세부, 미기재 하이퍼파라미터 등)을
  사용자가 직접 채워야 합니다.

---

## 4. 컴퓨트 환경

- 학습 재현에는 GPU/CUDA와 상당한 실행 시간이 필요하지만, 저장소는 이를 프로비저닝하지 않습니다.
- 생성된 `reproduce.sh`는 conda/venv + pip 설치 수준만 가정합니다.

---

## 5. 평가가 "실험 결과 일치"를 검증하지 않음

- [`codes/eval.py`](../codes/eval.py)의 평가는 모델이 코드의 **충실도(faithfulness)를 1~5점**으로 채점하는 것이며,
  재현된 지표(정확도·BLEU 등)가 논문 수치와 맞는지 **실행해서 검증하지 않습니다.**
- `ref_based` 평가를 쓰려면 저자 공식 코드(`--gold_repo_dir`)를 **별도로 확보**해야 합니다.

---

## 6. (별개) Paper2Code 논문 자체의 벤치마크 재현

논문 표의 수치를 재현하려는 경우 추가로 필요한 것:

- [`data/paper2code/paper2code_data.zip`](../data/paper2code) 압축 해제
- (PDF/JSON 쌍이 필요하면) 구글드라이브의 `paper2code_full_data.zip` 별도 다운로드
- PDF→JSON 변환용 [s2orc-doc2json](https://github.com/allenai/s2orc-doc2json) (Java/GROBID 필요)
- 논문의 **베이스라인·PaperBench 비교 설정은 이 저장소에 포함돼 있지 않음**

---

## 요약 표

| 실제 실험 재현에 필요한 것 | 저장소 제공 여부 |
|---|---|
| 코드 생성 | ✅ (`run.sh`) |
| 실행 스크립트(`reproduce.sh`) 생성 | ⚠️ 별도 실행 ([3.1_coding_sh.py](../codes/3.1_coding_sh.py)) |
| 실행 → 에러 → 자동 디버깅 루프 | ⚠️ 수동 오케스트레이션 + [버그 수정 필요](../codes/4_debugging.py#L138) |
| 실제 학습 데이터셋 | ❌ 직접 확보 |
| 미기재 하이퍼파라미터 | ❌ 직접 보완 |
| GPU/컴퓨트 | ❌ 직접 준비 |
| 실험 결과 수치 검증 | ❌ (평가는 코드 충실도만) |
| 공식 gold 코드 (ref_based용) | ❌ 직접 확보 |

**한 줄 결론**: PaperCoder는 *논문 → 실행 가능한 코드 초안*까지를 자동화하며,
그 코드를 실제로 돌려 논문 수치를 재현하는 것은
*생성 → reproduce.sh → 실행 → 디버깅 → 데이터·하이퍼파라미터 보완*의 반복 작업으로 사용자 몫으로 남습니다.
